# "법 바뀌었는데, 취업규칙은요?" — 그 질문을 코드로 끝내는 법

## 매년 반복되는 그 작업, 익숙하시죠?

법령이 개정됩니다. 고용노동부에서 알림이 오고, 법제처 홈페이지에서 신구조문을 찾아봅니다. 근로기준법, 남녀고용평등법, 산업안전보건법... **추적해야 할 법령만 10개가 넘습니다.**

그리고 시작되죠. 엑셀을 열고, 회사 취업규칙을 펼치고, 눈으로 하나하나 대조하는 작업.

"제60조가 바뀌었으니까... 우리 취업규칙 제15조 연차 조항을 고쳐야 하고... 신구조문 대조표도 만들어야 하고..."

**이 작업, 한 건당 최소 2~3시간입니다.** 관리하는 회사가 여러 곳이면? 법령 하나 바뀔 때마다 하루가 날아갑니다.

저도 그랬습니다. 매번 법제처 홈페이지를 뒤지고, Word에서 표를 만들고, 혹시 빠뜨린 조항이 없는지 불안한 마음으로 제출했습니다.

**그래서 만들기 시작했습니다. 코드로.**

---

## 발상의 전환: "법제처에 API가 있다고?"

많은 분들이 모르시는 사실이 하나 있습니다. **법제처는 국가법령정보를 API로 제공합니다.** 공공데이터포털(data.go.kr)에서 신청하면, 현행 법령의 이름, 시행일, 개정 구분까지 XML 데이터로 받아볼 수 있습니다.

이게 왜 중요하냐고요?

**사람이 눈으로 확인하던 것을, 코드가 자동으로 확인할 수 있다는 뜻입니다.**

```
[기존 방식]
법제처 접속 → 법령 검색 → 눈으로 확인 → 해당 조문 찾기 → Word에 정리

[자동화 방식]
python main.py check-updates --laws 근로기준법
→ 3초 만에 "근로기준법, 일부개정, 시행일 2025-10-23" 확인 완료
```

**3시간이 3초가 됩니다.** 과장이 아닙니다.

## 구축 과정: 삽질의 기록 (그리고 해결)

### 1단계. API 키 발급 — 생각보다 복잡한 인증의 세계

법제처 API에 접근하려면 두 가지 경로가 있습니다.

| 경로 | 인증 방식 | 특징 |
|------|----------|------|
| **공공데이터포털** (data.go.kr) | ServiceKey | 법령 목록 검색. 자동 승인 |
| **국가법령정보 공동활용** (law.go.kr) | OC (회원 ID) | 조문 상세 조회. 수동 승인 |

처음에는 law.go.kr만 사용하려 했는데, **OC 승인에 시간이 걸립니다.** 특히 연휴 기간에 신청하면 며칠이 소요될 수 있습니다.

그래서 **이중 구조로 설계했습니다.**

- 법령 검색(어떤 법이 바뀌었나?) → **공공데이터포털** (즉시 사용 가능)
- 조문 상세(어떤 조항이 바뀌었나?) → **law.go.kr** (승인 후 자동 전환)

```python
class LawAPIClient:
    # 공공데이터포털 (법령 검색)
    DATA_GO_KR_URL = "https://apis.data.go.kr/1170000/law/lawSearchList.do"
    # 국가법령정보 공동활용 (조문 상세 조회)
    LAW_GO_KR_URL = "http://www.law.go.kr/DRF/lawService.do"
```

**코드 3줄이지만, 이 설계 결정에 반나절이 걸렸습니다.** 하나의 API에 의존하면 장애 시 전체가 멈추니까요.

### 2단계. 인코딩 지옥 — 한국 공공 API의 단골 트러블

공공데이터포털 API의 ServiceKey에는 함정이 있습니다. 포털에서 발급하는 키에 `%2F`나 `%3D` 같은 **URL 인코딩된 문자**가 포함되어 있거든요.

Python의 `requests` 라이브러리는 파라미터를 자동으로 인코딩합니다. 이미 인코딩된 키를 또 인코딩하면? **이중 인코딩**으로 401 에러가 발생합니다.

```python
# 이렇게 하면 안 됩니다
requests.get(url, params={"serviceKey": encoded_key})  # 401!

# 이렇게 해야 합니다 — URL에 직접 삽입
url = f"{base_url}?serviceKey={encoded_key}&target=law&query={query}"
requests.get(url)  # 200 OK
```

**공공데이터포털 API를 쓰는 모든 개발자가 한 번씩 겪는 통과의례입니다.** 이 글을 읽는 분들은 이 삽질을 건너뛰세요.

### 3단계. 법령명의 함정 — `·`와 `ㆍ`는 다릅니다

"남녀고용평등과 일·가정 양립 지원에 관한 법률"을 검색하면 결과가 0건입니다.

왜? API에 등록된 법령명은 **"일ㆍ가정"** (가운뎃점, U+318D)인데, 우리가 보통 쓰는 건 **"일·가정"** (중간점, U+00B7)이기 때문입니다.

**눈으로는 같아 보이지만, 컴퓨터에게는 완전히 다른 문자입니다.**

이런 디테일이 노무 자동화의 난이도를 올리는 요소입니다. 법률 텍스트는 특수문자, 괄호 형식, 조문 번호 체계 모두 일반 텍스트와 다르거든요.

## 핵심 성과: 코드 한 줄로 10개 법령 추적

현재 자동 추적하는 법령 목록입니다.

| 법령 | 우선순위 | 취업규칙 영향 범위 |
|------|---------|-----------------|
| 근로기준법 | Critical | 근무시간, 휴가, 임금, 해고 |
| 최저임금법 | Critical | 임금 |
| 남녀고용평등법 | Critical | 모성보호, 육아휴직 |
| 산업안전보건법 | Critical | 안전보건, 건강진단 |
| 근로자퇴직급여 보장법 | Critical | 퇴직급여, 퇴직연금 |
| 기간제법 | High | 비정규직, 차별금지 |

```bash
$ python main.py check-updates --laws 근로기준법 --since 2024-01-01

📋 법령 개정사항 확인

  >> 근로기준법 확인 중...
  📌 근로기준법
     시행일: 20251023
     제개정: 일부개정
```

**이 한 줄이, 법제처 홈페이지를 열고 → 검색하고 → 시행일 확인하고 → 메모하는 5분을 대체합니다.** 법령 10개면 50분이 절약됩니다.

## 한글 파일(.hwpx)도 됩니다

노무사 실무에서 취업규칙은 대부분 **한글 파일**입니다. `.hwp`가 아직 표준이죠. 다행히 최신 한글 포맷인 `.hwpx`는 사실 **ZIP 파일 안에 XML이 들어있는 구조**입니다.

```python
# .hwpx = ZIP + XML (OWPML 형식)
with zipfile.ZipFile("취업규칙.hwpx", "r") as zf:
    xml_bytes = zf.read("Contents/section0.xml")
    # XML 파싱 → 조문 추출
```

별도의 한글 뷰어나 유료 라이브러리 없이도, **Python 기본 라이브러리만으로 취업규칙을 조문 단위로 파싱할 수 있습니다.**

`.docx`와 `.hwpx` 모두 지원하므로, 어떤 형식의 취업규칙이든 투입할 수 있습니다.

## 신구조문 대조표, 자동 생성

법령이 바뀌면 반드시 만들어야 하는 문서. **신구조문 대조표.**

| 조문번호 | 현행 규정 | 변경 규정 | 변경 사유 |
|---------|----------|----------|----------|
| 제15조 | 연차유급휴가는... | 연차유급휴가는... (단, 입사 1년 미만...) | 근로기준법 제60조 개정 |

이 표를 Word로 만들면 서식 맞추느라 시간이 걸립니다. **Excel(.xlsx)로 생성하면 데이터 관리와 필터링이 훨씬 편합니다.**

```python
from src.output.comparison_table import ComparisonTableGenerator

gen = ComparisonTableGenerator()
gen.generate(changes, "output/신구조문대조표.xlsx", company_name="주식회사 OO")
```

**맑은 고딕 폰트, 헤더 색상, 셀 테두리까지 자동 적용됩니다.** 열고 바로 출력해도 될 수준입니다.

## 아직 남은 과제 — 솔직하게

이 시스템이 아직 완벽하지는 않습니다.

1. **조문 상세 조회**: law.go.kr OC 승인이 나야 개별 조문 수준의 비교가 가능합니다. (현재 승인 대기 중)
2. **AI 매칭**: "근로기준법 제60조 개정 → 취업규칙 제15조 영향" 이 판단은 아직 구현 전입니다. Phase 3에서 AI를 활용할 계획입니다.
3. **실제 취업규칙 테스트**: 다양한 회사의 취업규칙으로 파싱 정확도를 검증해야 합니다.

**하지만 중요한 건, 뼈대가 세워졌다는 겁니다.** API 연동이 끝나면 나머지는 살을 붙이는 작업입니다.

## 노무사가 코드를 쓴다는 것

"노무사가 왜 코딩을 해요?"

이 질문을 많이 받습니다. 제 대답은 이렇습니다.

**"코딩을 하는 게 아니라, 업무 프로세스를 재설계하는 겁니다."**

법령 개정 확인 → 취업규칙 대조 → 수정안 작성 → 신구조문 대조표 생성. 이 흐름은 10년 전이나 지금이나 똑같습니다. 달라진 건, **이 흐름을 자동화할 도구가 생겼다는 것** 뿐입니다.

엑셀 함수를 쓸 줄 아는 것이 경쟁력이던 시대가 있었습니다. 지금은 **API를 연결하고, 데이터를 파싱하고, 문서를 자동 생성할 수 있는 것**이 경쟁력입니다.

코드를 직접 쓰지 않더라도 괜찮습니다. **이런 도구가 존재한다는 것을 아는 것만으로도**, 업무를 바라보는 눈이 달라집니다.
